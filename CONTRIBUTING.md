# Contributing to CodeNotes

Thank you for your interest in contributing to CodeNotes! This document outlines how to set up your development environment, understand our conventions, and safely extend the extension.

---

## 1. Development Setup

### Prerequisites
- **Node.js**: v18.0.0 or higher
- **npm**: v9.0.0 or higher
- **VS Code**: v1.85.0 or higher

### Initializing the Workspace
1. Fork the repository and clone it locally.
2. Open the `codenotes-extension` directory in VS Code.
   ```bash
   cd CodeNotes/codenotes-extension
   ```
3. Install dependencies:
   ```bash
   npm install
   ```

---

## 2. Build Process

The extension consists of two separate build targets: the **Extension Host** (Node.js) and the **Webview** (Browser).

- **`npm run compile`**: Compiles both targets. It runs `tsc` for the host and `esbuild` for the webview bundle.
- **`npm run compile:host`**: Compiles only the Extension Host.
- **`npm run compile:webview`**: Bundles `src/draw/webview/main.ts` into `out/webview/main.js`.
- **`npm run watch`**: Watches and recompiles the Extension Host. **Note:** There is no watch mode for the webview. You must manually run `npm run compile:webview` after making changes to the webview code.

---

## 3. Project Architecture (TL;DR)

For a deep dive, please read **[ARCHITECTURE.md](ARCHITECTURE.md)**.

- **Host & Webview Boundary**: The Extension Host manages file I/O (`.codenotes/`), commands, and decorations. The Webview handles the Draw Mode canvas UI. They communicate exclusively via a typed message passing protocol defined in `drawTypes.ts`.
- **Custom Editor**: Draw Mode is a `CustomReadonlyEditorProvider`. It does not hijack the live editor; it renders a read-only, syntax-highlighted copy of the code using `highlight.js`, overlaid with a 2D canvas.
- **Persistence**: Data is saved to `.codenotes/drawings/drawings.json` (drawings) and `.codenotes/notes.json` (hover notes). Images are saved as separate files in `.codenotes/drawings/assets/`.

---

## 4. Coding Conventions

- **TypeScript Strict Mode**: The project enforces strict typing. Do not use `@ts-ignore` or `any` unless absolutely necessary (and leave a comment explaining why).
- **No Direct DOM Hacks**: Do not attempt to inject CSS or HTML into the native VS Code editor DOM. Use only supported VS Code APIs (Decorations, Hovers, Webviews, Custom Editors).
- **Security / CSP**: The webview uses a strict Content Security Policy (CSP) with a one-time nonce. No external scripts, remote stylesheets, or remote images are permitted. Do not modify the CSP to bypass this.
- **Document-Space Coordinates**: All coordinates (`x`, `y`, `width`, `height`) are stored in document space (zoom scale 1.0). Never store screen-space pixels in the `ObjectStore`.

---

## 5. Pull Request Workflow

1. Create a feature branch from `main`: `git checkout -b feat/my-new-feature`
2. Make your changes, ensuring you run `npm run compile` to catch any TypeScript or esbuild errors.
3. Test your changes manually in the Extension Development Host (press **F5** in VS Code).
4. Commit your changes with clear, descriptive commit messages.
5. Push to your fork and open a Pull Request against the main repository.

---

## 6. Testing Requirements

We currently rely on manual verification before merging. When submitting a PR, please test the following depending on what you changed:

- **Hover Notes**: Add a note, edit it, delete it. Edit the source file and ensure the note correctly re-anchors to the new line position (content-based anchoring).
- **Draw Mode**: 
  - Ensure pen pressure works (if you have hardware support) or falls back smoothly (mouse).
  - Verify palm rejection is fully functional (touch events ignored while pen is active).
  - Test Undo/Redo (`Ctrl+Z`, `Ctrl+Shift+Z`) for your new tool/change.
  - Verify image pasting/dropping correctly downscales large images (>2000px).
  - Ensure the "Save failed" toast appears if the file cannot be written.

---

## 7. Debugging

### Debugging the Extension Host
1. Press **F5** to launch the **Extension Development Host**.
2. Breakpoints placed in `src/*.ts` (outside the `webview` folder) will be hit in your main VS Code window.
3. Console output (`console.log`) from the host appears in the **Debug Console** of the main window.

### Debugging the Webview
Because the webview runs in a separate Chromium instance, standard breakpoints won't work out of the box.
1. Open the Command Palette (`Ctrl+Shift+P` / `Cmd+Shift+P`) in the Extension Development Host window.
2. Run **`Developer: Open Webview Developer Tools`**.
3. This opens a Chrome DevTools inspector attached specifically to the CodeNotes webview.
4. You can inspect the DOM, view network requests, and read `console.log` output generated by `src/draw/webview/*.ts` files.

## 8. How to Add New Tools Safely

If you are adding a new drawing tool (e.g., a Highlighter or Polygon tool), follow this strict pattern to ensure it integrates correctly with the canvas engine, undo history, and palm rejection systems.

### Step 1: Create the Tool Class
Create a new file in `src/draw/webview/` (e.g., `myNewTool.ts`). Your class must implement `attach(canvas)` and `detach(canvas)`.

```typescript
import type { CanvasEngine } from "./canvasEngine";

export class MyNewTool {
  private activePointerId: number | null = null;
  private activePenPointerId: number | null = null; // Required for palm rejection

  constructor(private readonly engine: CanvasEngine) {}

  attach(canvas: HTMLCanvasElement): void {
    canvas.addEventListener("pointerdown", this.onPointerDown);
    // ... pointermove, pointerup, pointercancel
  }

  detach(canvas: HTMLCanvasElement): void {
    canvas.removeEventListener("pointerdown", this.onPointerDown);
    // ... remove others, clean up state
  }
}
```

### Step 2: Implement Palm Rejection
CodeNotes strictly enforces TRD §6.3. Every tool MUST include this guard in its `pointerdown` handler:

```typescript
private readonly onPointerDown = (e: PointerEvent): void => {
  if (!e.isPrimary) return;

  // Palm Rejection: Ignore touch if a pen is actively drawing
  if (e.pointerType === "touch" && this.activePenPointerId !== null) return;
  
  // Track the pen pointer
  if (e.pointerType === "pen") {
    this.activePenPointerId = e.pointerId;
  }

  this.activePointerId = e.pointerId;
  (e.target as HTMLCanvasElement).setPointerCapture(e.pointerId);
  // ... begin tool logic
};
```
*Don't forget to clear `activePenPointerId` in `pointerup` and `pointercancel`.*

### Step 3: Integrate with ObjectStore and History
When your tool completes an action (usually in `pointerup`), it must:
1. Snapshot the store **before** the mutation.
2. Mutate the `ObjectStore` (`this.engine.getStore().add(...)`).
3. Snapshot the store **after** the mutation and push to `HistoryManager`.
4. Trigger a save and a UI redraw.

```typescript
private readonly onPointerUp = (e: PointerEvent): void => {
  if (e.pointerId !== this.activePointerId) return;

  const store = this.engine.getStore();
  const history = this.engine.getHistory();
  
  const beforeSnapshot = store.snapshot();
  
  // Create your object (ensure it satisfies a DrawObject type)
  store.add({
    id: randomUUID(),
    layerId: this.engine.activeLayerId,
    // ...
  });

  // Record history
  history.pushAction({ before: beforeSnapshot, after: store.snapshot() });

  // Queue a debounced save to disk
  this.engine.onChanged?.();

  // Force the canvas to re-render the committed objects
  (this.engine as any).redrawCommitted();
};
```

### Step 4: Register the Tool in `main.ts`
1. Instantiate your tool in `initCanvas()`.
2. Add it to the toolbar HTML.
3. Update the `selectTool()` switch statement to handle attaching/detaching your tool correctly.
